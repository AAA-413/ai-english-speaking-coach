# Demo Video Script

建议时长：4-5 分钟

视频目标：让评委清楚看到这是一个英语口语练习工具，不只是聊天页面；同时解释原本的实时语音架构、豆包 RTC 当前问题，以及为什么使用 DeepSeek + TTS 作为稳定兜底演示链路。

## 0:00-0:30 开场

大家好，我们做的是一款 AI 英语口语练习工具，目标是帮助用户在真实场景里练习开口表达。

这个项目支持三类核心场景：面试、点餐和会议。用户选择场景后，AI 会扮演面试官、服务员或同事，和用户进行英文对话。练习结束后，系统会给出语法和表达纠错、综合评分、下一步练习建议，以及 scripted 跟读发音评测。

## 0:30-1:20 原本设计的架构

我们的架构设计不是把所有能力都塞进一个模型里，而是拆成两条链路。

第一条是实时对话链路，目标是自然、低延迟、尽量像真人对话。原本设计里，浏览器通过 WebRTC 接入端到端实时语音模型。用户的麦克风音频进入 RTC 房间，实时语音模型听到用户的话后，直接生成语音回复，浏览器订阅远端音频流并播放。

第二条是学习反馈链路，目标是稳定和准确。练习结束后，我们把对话 turns 交给文本模型做课后总结，包括分项评分、语法纠错、表达优化和下一步练习任务。发音评测则放在 scripted 跟读里，因为参考文本是确定的，评分会比开放对话里的实时转写更可靠。

这样拆分的原因是：实时对话最重要的是不卡顿、不频繁打断；而纠错和评分最重要的是准确、可解释、可量化。

## 1:20-2:05 豆包 RTC 当前问题和兜底策略

我们接入了豆包 O2.0 端到端实时语音模型，也实现了火山 RTC 相关链路：后端可以生成 RTC client token，可以签名调用 `StartVoiceChat`，结束时也可以调用 `StopVoiceChat`。浏览器侧已经可以加载火山 RTC Web SDK，加入同一个房间，并发布本地麦克风音频。

但是本地 QA 时遇到一个关键问题：`StartVoiceChat` 返回成功，浏览器也成功进房并发布麦克风，但事件日志里没有收到豆包 agent 的远端音频发布事件，也就是没有 `remote_publish`，所以前端没有远端音频流可以订阅和播放。

为了保证最后提交的视频能稳定展示完整产品闭环，我们没有把 demo 卡死在 RTC provider 配置问题上，而是保留豆包 RTC 代码和诊断日志，同时把默认演示链路切到稳定兜底：

用户输入英文回答后，后端调用 DeepSeek 生成短的场景化英文追问；再通过本地/server TTS 生成 AI 语音播放；最后仍然进入课后总结和发音评测流程。

所以这版 demo 展示的是完整口语训练产品闭环，豆包 RTC 是已集成但还需要继续和 provider 配置对齐的实时语音主线。

## 2:05-3:20 功能演示

现在看页面。

左侧可以选择练习场景。我这里选择“面试”，水平是 B1，纠错时机选择“课后总结纠错”。实时语音默认是“稳定实时语音”，也就是刚才说的 DeepSeek + TTS 兜底链路。

点击“开始练习”。AI 教练会先给出面试开场问题，并播放语音。

如果当前浏览器允许语音识别，用户可以直接说英文；如果浏览器不允许自动语音识别，也可以在输入框里输入英文回答。这里我输入：

```text
I have worked as a product manager for three years.
```

点击发送后，AI 会根据面试场景继续追问，比如让用户讲一个具体项目或者挑战。这个回复不是固定 mock，而是调用 DeepSeek 根据当前场景和对话上下文实时生成的。

这里页面上也保留了事件日志。我们可以看到 session start、server TTS、turn assistant、turn user 等事件。如果使用豆包 RTC 路线，也会在这里看到 RTC join、publish、StartVoiceChat、remote publish 等诊断信息。

## 3:20-4:20 课后总结和发音评测

练习几轮后点击“结束练习”。系统会生成课后反馈报告。

报告里有总分和多个分项，例如流利度、语法、词汇、互动自然度和发音相关指标。下面会列出用户原句、修改后的句子、中文解释，以及更自然的表达方式。

这部分的设计重点是让用户知道自己哪里错、为什么错、下次应该怎么说，而不是只给一个分数。

再往下是跟读发音评测。系统会推荐一句和本次场景相关的英文句子。用户点击录音，读出句子，再点击评分。后端会根据参考文本和识别结果给出 pronunciation、accuracy、fluency、completeness、prosody 等维度反馈。

如果没有配置第三方发音评测 key，系统会降级到 scripted scoring，保证 demo 和学习闭环不断掉。

## 4:20-4:50 工程亮点

工程上我们做了几个关键点。

第一，provider 是可替换的。实时语音可以走 OpenAI Realtime、豆包 O2.0，或者稳定兜底链路；总结和纠错可以走 DeepSeek 或 OpenAI；发音可以走 Azure 或 scripted fallback。

第二，所有关键事件都有诊断日志，方便定位到底是 token、RTC join、麦克风权限、远端音频发布，还是模型回复出了问题。

第三，实时对话和学习反馈分离，既能保证练习时的自然度，又能在课后给出更可靠的纠错和量化反馈。

## 4:50-5:00 结尾

总结一下，这个项目已经完成了英语口语练习的核心闭环：场景选择、AI 对话、语音播放、语法表达纠错、课后总结和发音跟读评测。

豆包 RTC 端到端语音是我们原本的实时主线，服务端启动和浏览器进房已经打通，但当前 provider 侧没有发布远端音频流，所以提交 demo 使用 DeepSeek + TTS 作为稳定兜底，保证评委能看到完整产品能力。

后续如果继续迭代，我会优先完成豆包 agent 远端音频发布问题的排查，然后把默认链路切回真正的端到端实时语音。

## 录制操作清单

1. 打开 `http://localhost:3000/`。
2. 选择“面试”场景。
3. 确认实时语音为“稳定实时语音（推荐）”。
4. 点击“开始练习”。
5. 如果语音没有自动播放，点击“播放 AI 语音”。
6. 在输入框输入英文回答并发送。
7. 展示 AI 追问和事件日志。
8. 点击“结束练习”。
9. 展示课后报告、纠错、分项评分。
10. 展示跟读发音评测入口。
11. 结尾说明豆包 RTC 当前问题和 DeepSeek 兜底策略。
